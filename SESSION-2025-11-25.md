## Implementation Details

### Backend Changes

**1. Enhanced `backend/app/analyse_history.py`**
- Made `get_best_image_for_period()` function more flexible with configurable parameters
- Added function documentation
- Made month, day_start, day_end configurable (not just hard-coded)
- Script can download missing historical satellite data for any year

**2. Created `/api/zones/{zone_id}/history` endpoint** (`backend/app/main.py`)
- Returns year-over-year health data for specified time period
- Default: mid-September (Sept 10-25) for years 2015-current
- Configurable query parameters:
  - `start_year`: Starting year (default: 2015)
  - `end_year`: Ending year (default: current year)
  - `month`: Target month (default: 9 = September)
  - `day_start`: Start of target window (default: 10)
  - `day_end`: End of target window (default: 25)
- Returns JSON with:
  - Health score, NDVI, NDMI for each year
  - Trend indicators (improving/stable/declining/baseline)
  - Trend icons (↗️/➡️/↘️/⏺️)
  - Handles missing data gracefully

**Trend Calculation Logic:**
- **Improving** (↗️): Health score increased by 5+ points
- **Stable** (➡️): Health score changed by -5 to +5 points
- **Declining** (↘️): Health score decreased by 5+ points
- **Baseline** (⏺️): First year with data (no comparison)

## Technical Decisions

1. **API Design**: Used query parameters for flexibility (year range, month, day window)
   - Allows future use cases beyond mid-September analysis
   - Defaults match the original analyse_history.py script requirements

2. **Frontend Component**: Chose bar chart over line chart
   - Better for year-over-year comparison with discrete data points
   - Color-coding makes trends immediately visible
   - Works well with missing data (gaps don't break visualization)

3. **Trend Calculation**: Used ±5 point threshold for significant changes
   - Matches the thresholds in original analyse_history.py script
   - Avoids noise from small fluctuations

4. **Data Loading**: Parallel API calls for health, alerts, and history
   - Improves page load performance
   - All data loads simultaneously when zone is selected

5. **Historical Script Path**:
   - In Docker container, files are at `/app/app/analyse_history.py`
   - Correct command: `docker-compose exec processor python app/analyse_history.py`

## Lessons Learned

1. **Docker Path Confusion:** Container paths differ from host paths
   - Host: `/volume1/docker/olive-monitoring/backend/app/`
   - Container: `/app/app/` (because docker-compose mounts `./backend` as `/app`)

2. **Frontend Deployment:** Always rebuild after React code changes
   - Development builds (npm run dev) show changes immediately
   - Production builds (npm run build) require explicit rebuild step
   - nginx serves static files from `dist/`, not `src/`

3. **API Container Restart:** Code changes in backend require container restart
   - `docker-compose restart api` for code changes
   - `docker-compose up -d --build api` for dependency changes

4. **Historical Data Processing:** Can take significant time
   - ~5-10 minutes per year of missing data
   - Downloads ~300-500 MB per image
   - Consider running during off-hours for large date ranges
